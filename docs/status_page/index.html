01 The Experiment

What this is, and what it isn't

This is an experiment, and we don't know how it ends. The goal: take the original Star Trek: Bridge Commander — a 2002 Windows-only game — and rebuild it's closed-source engine from scratch so the original game scripts run again on modern macOS, Linux, and Windows.

The original development team had a great vision, pushing the boundaries of Python scripting to create an ambitious space simulation game which captured the scale and scope of Star Trek. 24 years we are being equally ambitious, attempting to bring these game scripts back to life with the power of AI.

What makes this experiment unusual is the method. Almost every line of code in this project was written with Claude (Anthropic's LLM) as a pair-programmer. That includes the NIF asset loader, the C++ renderer, the host loop that embeds CPython, the physics integration, the shaders — the lot. As far as we're aware, this is the first serious attempt to reconstruct a legacy game engine of this size with an LLM as the principal collaborator.

It might not work. It might stall at Phase 05 with the AI scripts running fine but combat refusing to feel right. It might hit a wall at character animation. It might surprise me and land. The point of putting this page up is to make the attempt visible — what's been built, what's next, what broke last week.

If you are reading this and thinking "that's not how you rebuild a game engine," you're probably right. Maybe it will work though! Our progress will be honest.

02 Mission

What we're building

We're building a modernised version of the Bridge Commander game engine, with Claude Code as our pair-programmer. The aim is simple: take a beloved 2002 game that only ever ran on Windows, and bring it back to life on the machines people actually use today — without losing what made it special.

The original team left behind a remarkable gift. Every mission, every ship, every line of dialogue lives in plain, readable scripts they released as an SDK. The game logic still works. What's missing is the engine underneath — the part that draws the ships, plays the sounds, and ties it all together. That's what we're rebuilding.

Faithful to the original

The original missions, ships, and scripts run as-is. We're rebuilding the engine, not the game — every mission you remember should play the way you remember it.

Modern visuals

A brand-new graphics engine, built from scratch, drawing the same ships and effects but with the headroom to look great on today's displays.

Runs everywhere

macOS first, Linux next, Windows for the 1.0 release. No emulators, no compatibility layers — for the first time, Bridge Commander will run natively beyond Windows.

Open and community-friendly

Every commit is public. The fans who kept Bridge Commander alive for two decades are the natural home for this — and mod support is firmly on the roadmap.

03 Latest Signal
Stardate 2026-05-18 · Where we are

The core blocks of the game are coming together. The original SDK loads. Star systems and the things inside them — ships, planets, stations — populate a working scene. Ships load with their subsystems intact. You can fly the player ship around, and there's early support for weapons fire and shield impacts. First steps toward the bridge interior are in too.

One of the surprises has been just how much the original SDK gives us. Every script is a hint about how the 2002 team handled a problem — a quiet expert sitting in the repo. Claude is good at reading those hints once you point it at them, though it sometimes needs a nudge to go and look first rather than guessing.

Two things keep biting us. The first is testing against the original Windows game. We can run it, but the only way to get data out is by abusing the config-file system the game uses for its options menu — a narrow, fragile channel that often crashes the game before we get our answer.

The second is units and scale. The original engine assumed everything had been exported to a consistent size, and quietly fixed up anything that wasn't. So when we load the same assets fresh, planets come out smaller than ships, and weapon hardpoints sit slightly off the hulls they're meant to bolt onto. Each one is a small puzzle in what convention the original code was silently relying on.

Next up: We'll be starting work to restore Bridge Commander's NPC ship AI systems, enabling non player ships to move and even attack.

05 Under the Hood

Technical stack

Host process
C++ executable embedding CPython 3 — drives the BC Python game loop natively, no shim DLL.
Renderer
In-house C++ renderer parsing BC's NIF asset blocks. Passes: opaque hull, sovereign shields, glow (via AddLOD), space dust.
Physics
PyBullet during the Phase 1 headless work; transitioning to native Bullet through the C++ host as Phase 2 settles.
Audio
OpenAL — stands up in Phase 06 alongside the audio script bindings.
UI
RmlUi for HUD overlays; reusable component library mirrors the original BC LoadInterface scripts.
Asset pipeline
BC NIF files (NetImmerse / Gamebryo blocks). Right-handed coordinate space, W-first quaternions, link-ID streaming.
Platforms
macOS-first development. Linux follows. Windows joins for 1.0. Single codebase, no DirectX dependency.
Source material
BC SDK v1.1 — 1,228 Python files, fully readable, unchanged since 2002.

How Claude fits

The collaboration model in plain English: a human (Mark) directs, reviews, and makes architecture calls. Claude writes the bulk of the C++, shaders, build scripts, and tooling. Each non-trivial change goes through a brainstorm → spec → plan → implementation loop, with the spec committed to docs/superpowers/specs/ before code is written. Every commit is human-reviewed before merge.

The repository is structured to make this auditable: design specs are first-class artefacts, surveys of the original game's behaviour live alongside the code, and the project's CLAUDE.md is itself documentation for both human and AI readers.
