# Bridge Commander Engine Reimplementation
# Gap Analysis

**Date:** May 2026  
**Source:** BC SDK v1.1 (bcsdk1_1.zip); full game installation  
**Scope:** Analysis of all engine gaps requiring resolution before or during
reimplementation, based on static analysis of the Python source, SWIG-generated
`App.py`, hardpoints files, SDK documentation, SDK tutorial missions, and full
static analysis of all 1228 SDK source files.

**OQ status key:** ❌ Open — ✅ Answered — ⚠️ Partial

---

## Executive Summary

The SWIG-generated `App.py` is the defining discovery of this analysis. Rather
than a black box C++ engine with an opaque Python interface, the SDK provides a
complete, readable specification of every engine call the game makes. Every gap
identified has a known solution path. No gap requires novel engineering. No gap
proved intractable.

**Recommended implementation phases:**

- **Phase 1 — Headless logic engine:** Replace `Appc` with a Python
  reimplementation, physics via PyBullet, event system from scratch, no
  renderer. Sufficient to run tutorial missions and validate all game logic.
  Estimated 3–6 months for a small focused team.

- **Phase 2 — Full engine:** OpenMW NIF loader extended with NifSkope
  definitions, OpenAL audio, full 3D rendering, character animation.
  Estimated 18 months to 2 years for the same team.

## Gap 1 — Appc C Extension Interface

### Summary

`App.py` is auto-generated by SWIG and wraps every engine call through
`Appc` — a compiled C extension DLL. Every class, method, and property
crossing the boundary is documented in `App.py`. The signatures are all
visible. What requires work is implementing the semantics — what each
call actually computes.

### What we know

- Complete API surface is readable in `App.py` without ambiguity
- SWIG wrapper pattern means every engine call follows the same
  predictable dispatch: `App.SomeClass.SomeMethod()` →
  `Appc.SomeClass_SomeMethod(self.this, ...)`
- The entire interface can be intercepted by replacing `Appc` with a
  logging shim before anything else loads — no native tooling required
- Object hierarchy is documented in `basics.html`:
  `ObjectClass → PhysicsObjectClass → DamageableObject → ShipClass`

### Solution path

Replace `Appc` with a pure Python module implementing the same interface.
Instrument the running game with an `Appc` logging shim to capture
runtime argument values and call sequences for semantics validation.

### Open questions requiring investigation

**OQ-1.1 — Per-method semantics for less-used API calls** ✅  
Full static analysis of all 1228 SDK source files resolves this. The SDK
contains complete mission scripts, AI scripts, and all ship/hardpoints
definitions. Every non-trivial API call has at least one call site with
visible argument patterns. Remaining ambiguities (null handling, error
states) are best resolved by running the tutorial missions against the
reimplemented shim and observing failures, not by pre-emptive instrumentation.

**OQ-1.2 — Object lifecycle and memory management** ✅  
`__del__` is defined in exactly 4 condition classes. `ConditionInRange.__del__`
is the canonical example: it explicitly removes the proximity sphere from the
engine (`pProxSphere.RemoveFromSet()`). The pattern is consistent — Python
owns Appc objects it creates via constructor, and must explicitly clean them
up via engine calls in `__del__`. No `thisown` manipulation is needed from
the Python side; the SWIG `thisown` flag is set by the C++ factory methods,
not by Python code.

**OQ-1.3 — Serialisation format for save/load** ✅  
39 classes define `__getstate__` and `__setstate__`. The pattern is uniform:
`__getstate__` returns a plain dict of Python-side state (object names,
configuration values), discarding any Appc handles. `__setstate__` re-looks
up live Appc objects by name and re-attaches handlers. `cPickle` is imported
in `Autoexec.py`. No novel serialisation contract is required — the pattern
is the same across all 39 implementations.

## Gap 2 — Physics Engine

### Summary

Ship physics parameters are fully specified in plain Python source.
`GlobalPropertyTemplates.py` contains mass and rotational inertia for
every ship class. Hardpoints files contain `MaxAccel`, `MaxAngularAccel`,
`MaxAngularVelocity`, and `MaxSpeed` for every ship. No empirical
calibration is required — the spec is complete.

### What we know

- **`GlobalPropertyTemplates.py`:** `SetMass()`, `SetRotationalInertia()`
  per ship class
- **Hardpoints files:** `SetMaxAccel()`, `SetMaxAngularAccel()`,
  `SetMaxAngularVelocity()`, `SetMaxSpeed()` per ship
- Space combat physics is zero-gravity rigid body dynamics — a light
  workload with no joints, cloth, or soft bodies
- Maximum ~30 simultaneous physics objects in any scene
- PyBullet is the recommended replacement — Python-native, handles
  zero-gravity rigid body dynamics, parameters map directly from source

### Solution path

PyBullet with parameters loaded directly from `GlobalPropertyTemplates.py`
and hardpoints files. Disable default gravity. Map `SetMaxAccel` to
PyBullet force limits.

### Open questions requiring investigation

**OQ-2.1 — Engine condition degradation formula** ⚠️  
`SetDisabledPercentage()` values are now known: 0.0 for hull (cannot be
disabled), 0.75 for shields, sensors, power plant, and weapons — defined
as constants in each ship's hardpoints file at creation time, not computed
dynamically. The value appears to be the fraction of effectiveness lost at
0% health (so 0.75 → operates at 25% capacity when fully disabled). The
actual scaling function — how the engine interpolates between full health
and the disabled floor — is still engine-side and requires instrumentation.

**OQ-2.2 — Warp physics model**  
Warp travel moves ships between sets rather than integrating physics
across large distances. The transition mechanics — how velocity is
handled at warp entry/exit, how the stretched warp visual model relates
to collision radius — are engine-side. Approach: instrument warp
entry/exit sequences with position and velocity logging.

**OQ-2.3 — Tractor beam physics**  
`TractorBeamProperty` exists in hardpoints but the force model applied
to tethered objects is engine-side. Approach: instrument tractor beam
sessions, log relative positions and velocities of towing and towed
ships over time.

## Gap 3 — Gamebryo / NIF Renderer

### Summary

The game uses NetImmerse/Gamebryo with `.nif` files for all models. The
NIF format is versioned and game-specific extensions exist. Two open
source projects together provide a strong foundation: OpenMW (working
OSG-based NIF renderer) and NifSkope (BC-specific NIF format definitions
and BC named explicitly in its compatibility list).

### What we know

- NifSkope has explicit Bridge Commander support — BC NIF version and
  custom property types are already documented in `nif.xml`
- OpenMW provides a working OpenSceneGraph-based NIF renderer covering
  the core Gamebryo scene graph (`NiNode`, `NiTriShape`,
  `NiTexturingProperty`, skin/bone hierarchies)
- BC Python source defines rendering conventions explicitly:
  `_glow` suffix for glow maps, `_specular` suffix for specular maps,
  `SpecularCoef` per ship class
- LOD system is specified in ship scripts: `AddLOD()` with switch
  distances and damage resolution values
- Procedural effects (shield bubbles, phaser beams, engine trails) are
  defined in `Tactical/EffectTextures.py` and
  `Tactical/Projectiles/` — spec is readable

### Solution path

OpenMW NIF loader as base, extended with BC-specific block types
identified via NifSkope `nif.xml`. Procedural effects reimplemented
from Python source spec. Validate against actual BC assets.

### Open questions requiring investigation

**OQ-3.1 — BC NIF version compatibility with OpenMW loader**  
OpenMW targets Morrowind NIF v4.0.0.2. BC may use a different version
or custom extensions not covered by OpenMW's loader. Approach: run
actual BC NIF files through OpenMW loader, identify which block types
fail, cross-reference with NifSkope definitions.

**OQ-3.2 — Surface and internal damage geometry tagging**  
Hardpoints reference `SetDamageResolution()` suggesting geometry is
tagged for damage visualisation. How damage geometry is marked in the
NIF files is not visible from Python source. Approach: inspect BC NIF
files in NifSkope, identify damage-related node naming conventions.

**OQ-3.3 — Character body/head NIF assembly**  
`CharacterClass_Create()` takes separate body and head NIF paths and
assembles them at runtime. How the skeleton is merged and how
`ReplaceBodyAndHead()` swaps textures on the assembled model is
engine-side. Approach: inspect character NIF files in NifSkope,
examine skeleton node structure.

## Gap 4 — Event System

### Summary

The event system's complete public surface is visible. All ~100 `ET_`
event type constants are enumerated in `App.py`. Two dispatch mechanisms
are documented (`AddPythonFuncHandlerForInstance` and
`AddBroadcastPythonFuncHandler`). Two timer managers exist (simulation
time and real time). The SDK documentation (`makingamission.html`)
explicitly documents event structure and the five mission flow mechanisms
(events, timers, sequences, conditions, proximities).

### What we know

- Complete `ET_` event type catalogue visible in `App.py`
- Event structure documented: type, source object, destination object,
  optional auxiliary data depending on event class
- `g_kEventManager` — global broadcast dispatch
- `AddPythonFuncHandlerForInstance` — per-object instance dispatch
- `g_kTimerManager` — simulation time timers
- `g_kRealtimeTimerManager` — wall clock time timers
- `App.Game_GetNextEventType()` and `App.g_kVarManager.MakeEpisodeEventType(N)` — two dynamic event type allocation paths (the latter used in multiplayer missions)
- Best-effort implementation of ~150–200 lines covers the vast majority
  of mission content; refinement via logged sessions for edge cases
- Tutorial missions in the SDK provide a concrete first test suite

### Solution path

Pure Python event manager implementing broadcast dictionary and instance
handler dictionary. Best-effort implementation first, refine against
logged sessions for specific failing scenarios.

### Open questions requiring investigation

**OQ-4.1 — Per-event-type payload schema** ✅  
Reconstructed from static analysis of all handler call sites. The complete
accessor set on `pEvent` is: `GetInt()`, `GetBool()`, `GetString()`,
`GetCString()`, `GetSource()`, `GetDestination()`, `GetFiringObject()`,
`GetPlacement()`, `GetUnicode()`, `GetKeyState()`, `GetMessage()`. Per-type
schema is now derivable by reading every handler in the SDK and noting which
accessors it calls. `Multiplayer/MissionShared.py` contains the most
comprehensive examples. No instrumentation needed for schema derivation.

**OQ-4.2 — Event dispatch ordering and reentrancy** ⚠️  
Reentrancy is confirmed: handlers in `MissionShared.py` create and fire new
events mid-handler via `App.TGEvent_Create()` followed by
`App.g_kEventManager.AddEvent()`. The chain-of-responsibility pattern
(`CallNextHandler(pEvent)`) is also confirmed. Whether `AddEvent` inside
a handler causes immediate dispatch or queues for end-of-frame is still
engine-side. Instrumentation with frame-accurate timestamps is needed to
distinguish the two models.

**OQ-4.3 — Handler priority and cancellation**  
Whether handlers can consume events to prevent other handlers seeing
them, and whether handler registration order affects invocation order,
is not documented. Approach: register multiple handlers for the same
event type, log invocation order across sessions.

**OQ-4.4 — Sequence execution model**  
Whether sequences block the caller, yield, or run asynchronously is
not specified. The `TGScriptAction` description explains what it does
but not the threading/callback model. Approach: instrument sequence
start, action completion, and sequence completion events with
frame-accurate timestamps.

## Gap 5 — Collision and Spatial Queries

### Summary

This gap is effectively closed from static analysis alone. The
`Conditions/` folder contains the full orchestration logic for all
spatial queries in pure Python. The engine provides only three
primitives underneath: `ProximityCheck`, `ProximityManager`, and
`GetLineIntersectObjects`. All have well-understood implementations.
Phaser arc geometry is fully specified in hardpoints files.

### What we know

- `ConditionInRange` — pure Python logic on `ProximityCheck` sphere;
  full source available including set-crossing, warp state, and group
  membership edge cases
- `ConditionInLineOfSight` — polls every 2 seconds; uses
  `GetLineIntersectObjects()` ray cast; documented as intentionally
  inefficient
- `ConditionInPhaserFiringArc` — polls every 0.5 seconds via
  `PythonMethodProcess`; calls `pBank.CanHit(vTargetLocation)` which
  is reimplementable from arc data in hardpoints
- Arc data fully specified: `SetArcWidthAngles()`,
  `SetArcHeightAngles()`, `SetOrientation()`, `SetPosition()` per bank
- `CanHit()` reimplementation: transform target to phaser bank local
  frame, check azimuth/elevation against arc min/max — ~15 lines of
  numpy
- `ProximityCheck` — bounding sphere attachment and inside/outside
  event firing; implementable via PyBullet ghost objects or distance loop
- `ProximityManager.GetLineIntersectObjects()` — ray-sphere intersection
  against set objects; standard 3D geometry

### Solution path

PyBullet or numpy for the three engine primitives. All orchestration
logic is already written in the Conditions folder.

### Open questions requiring investigation

**OQ-5.1 — ProximityCheck update frequency**  
`CheckProximity()` can be called explicitly to force an immediate check,
implying the proximity manager updates on a schedule rather than
continuously. The update frequency is unknown. Approach: instrument
proximity event fire times relative to frame numbers across sessions
with known object movements.

**OQ-5.2 — TorpedoTube arc geometry**  
`TorpedoTube.CanHit()` and `IsInArc()` are in `Appc` but torpedo tube
arc data follows the same pattern as phasers. Approach: read a torpedo
tube hardpoints definition and confirm the arc parameter pattern matches
the phaser implementation.

**OQ-5.3 — Warp radius during warp transition**  
`ConditionInRange` contains special handling for ships currently warping,
accessing `GetClonedModelRadius()` when warp state is not
`WES_NOT_WARPING`. The relationship between warp visual stretch and
effective collision radius is engine-side. Approach: instrument radius
reads during warp entry/exit sequences.

## Gap 6 — Audio System

### Summary

The `TGSound` API surface is completely visible and maps directly to
OpenAL concepts. The audio system is the most straightforward gap on
the list — no semantic complexity, perceptual correctness criteria
rather than mathematical ones. OpenAL via openal-soft is a direct
replacement.

### What we know

- Three sound categories: `LS_3D` (positional), `LS_STREAMED` (music),
  `LS_DELAY_LOADING`
- Three mix channels: SFX, Voice, Interface — independent volume control
- Full `TGSound` API: `Play`, `Stop`, `Pause`, `Rewind`,
  `PlayAndNotify`, `SetVolume`, `SetPitch`, `SetLooping`,
  `SetMinMaxDistance`, `SetConeData`, `AttachToNode`
- `PlayAndNotify` fires a completion event — used by dialogue sequences
  to chain voice lines; this is the one audio/event system integration
  point requiring reliable implementation
- `DynamicMusic.py` (13KB) suggests a sophisticated dynamic music system
  with transitions — separate concern from 3D SFX
- All sound assets are standard `.wav` files at known paths

### Solution path

OpenAL via openal-soft for all `LS_3D` and `LS_STREAMED` sounds.
`PlayAndNotify` completion callback maps to OpenAL source state
polling or AL_EXT_source_notifier. pygame.mixer as fallback for
streamed music if OpenAL streaming proves complex.

### Open questions requiring investigation

**OQ-6.1 — DynamicMusic transition model**  
`DynamicMusic.py` is 13KB suggesting non-trivial transition logic. How
music states map to gameplay states (combat, exploration, bridge) and
how crossfades are triggered is partially visible in Python but the
audio engine side of smooth crossfading is not. Approach: read
`DynamicMusic.py` fully; instrument music state changes during a
representative play session.

**OQ-6.2 — 3D audio attenuation model**  
`SetMinMaxDistance()` and `SetConeData()` configure spatial attenuation
but the specific attenuation curve (linear, inverse, inverse square) is
engine-side. OpenAL supports all three; the correct one needs to be
matched. Approach: instrument volume at known distances during a session
and fit the attenuation curve.

## Gap 7 — Game Loop and Frame Timing

### Summary

The game loop architecture is partially visible. Two independent time
streams exist (game time and real time). Time scale is a first-class
concept. `TimeSliceProcess` is the scheduling primitive with four
priority levels. `PythonMethodProcess` inherits from it. The loop is
single-threaded from Python's perspective. Four specific questions
require instrumentation to answer — all are low effort and should be
resolved before physics or timer implementation begins.

### What we know

- `g_kUtopiaModule.GetGameTime()` — simulation clock
- `g_kUtopiaModule.GetRealTime()` — wall clock
- `g_kUtopiaModule.SetTimeScale()` — scales game time relative to real
  time; used for cinematic slow motion
- `g_kTimerManager` runs on game time;
  `g_kRealtimeTimerManager` runs on real time
- `TGSystemWrapper` exposes `GetTimeSinceFrameStart`,
  `GetUpdateNumber` (frame counter), `GetTimeInSeconds`,
  `GetTimeElapsedInSeconds`
- `TimeSliceProcess` priorities: `UNSTOPPABLE`, `CRITICAL`, `NORMAL`,
  `LOW` — most Python usage is `NORMAL`
- `PythonMethodProcess` is the Python-callable process type, used by
  condition polling
- `sys.setcheckinterval(200)` in `Autoexec.py` — single-threaded,
  explicitly tuned for performance
- `UtopiaModule.Pause()` and `ForceUnpause()` — explicit pause/unpause
  distinct from time scale

### Solution path

Python-owned main loop driving physics, audio, and rendering subsystems
explicitly each tick. This inverts the original architecture (Python was
a guest in the engine's loop) but makes the loop deterministic and
testable — preferable for a TDD approach.

### Open questions requiring investigation

*(All four questions below are formally tracked and prioritised in*
*`open_questions.md`)*

**OQ-7.1 — Tick rate** ✅  
**60 Hz fixed.** Measured at 59.61 Hz (16.78 ms/tick) over 86.3s / 5145
ticks in Quick Battle. Theoretical 1/60 = 16.667 ms; deviation is within
measurement error. Physics time step = 16.667 ms. `TimeSliceProcess`
minimum polling granularity = 16.667 ms.

**OQ-7.2 — Subsystem update ordering within a tick**
*(Priority: High — resolve before AI integration)*  
Does physics integrate before or after AI runs? Do events fire before
or after physics? Approach: log `GetUpdateNumber` alongside physics
reads/writes, AI callbacks, event dispatch, and timer calls. Sort by
frame number to reconstruct within-tick ordering.

**OQ-7.3 — Time scale interaction with physics and AI** ⚠️  
*(Priority: Medium — resolve before cinematic implementation)*  
Baseline confirmed: time_scale = 0.98 ≈ 1.0 in normal Quick Battle
gameplay. Open question: does `SetTimeScale(0.5)` scale physics
delta-time, AI callback frequency, and `g_kTimerManager` timers while
`g_kRealtimeTimerManager` continues at wall speed? Requires a targeted
session triggering a cinematic slow-motion sequence.

**OQ-7.4 — TimeSliceProcess priority semantics** ✅  
Full source scan confirms: Python code uses only `NORMAL` and `LOW`. `LOW`
appears in exactly two scripts (`ConditionIncomingTorps`, `FriendliesInPlayerSetStronger`).
`CRITICAL` and `UNSTOPPABLE` have zero Python call sites — they are C++
internal priorities for rendering and physics. Reliable polling intervals
are safe to assume for all Python-visible processes. No instrumentation needed.
*(Also tracked as Q4 in `open_questions.md` — closed there too.)*

## Gap 8 — Character Animation System

### Summary

The animation system is less complex than feared. Lip sync uses texture
swapping rather than mesh deformation. Choreography reuses the existing
`TGSequence` system rather than being a separate subsystem. Skeletal
animation is standard NIF-based playback covered by the OpenMW/NifSkope
foundations identified for Gap 3. The only genuinely novel piece is
phoneme-driven lip sync, which is a polish item with good open source
solutions.

### What we know

**Layer 1 — Skeletal animation:** Pre-baked `.nif` animation files loaded
via `g_kAnimationManager.LoadAnimation()`. `TGAnimNode` plays named
animations by string key. `SetExclusiveAnimation`,
`SetNonExclusiveAnimation`, `BlendAnimation`, `SetBlendTime` provide
standard animation blending. Orchestration is pure Python.

**Layer 2 — Facial animation:** Lip sync and blinking via texture
swapping, not mesh deformation. `AddFacialImage("SpeakA", path)`,
`AddFacialImage("Blink0", path)` register textures for phoneme groups
and blink stages. Engine drives texture selection from audio amplitude
and timing. `SetBlinkStages(3)`, `SetBlinkChance(0.1)` configure
probabilistic blinking.

**Layer 3 — Sequence choreography:** `TGSequence` with `TGAnimAction`,
`CharacterAction`, `TGScriptAction`, `TGAnimPosition` — the same
sequence system used for mission scripting. No separate animation
choreography subsystem.

**Character state machine:** `CS_IDLE`, `CS_STANDING`, `CS_GLANCING`,
`CS_TURNED`, `CS_SEATED`, `CS_INITIATIVE` — fully enumerated. State
transitions driven by `CharacterAction` sequences completing and
firing `ET_CHARACTER_ANIMATION_DONE`.

**Random animations:** `SetRandomAnimationChance(0.75)` — probabilistic
idle animation selection on a timer. Fully reimplementable.

**Complete `CharacterAction` vocabulary:** `AT_SPEAK_LINE`,
`AT_SAY_LINE_AFTER_TURN`, `AT_PLAY_ANIMATION`, `AT_LOOK_AT_ME`,
`AT_GLANCE_AT`, `AT_BREATHE`, `AT_MENU_UP`, `AT_MENU_DOWN`,
`AT_BECOME_ACTIVE`, `AT_DISABLE_MENU`, `AT_ENABLE_INITIATIVE` and
others — all enumerated in `App.py`.

### Solution path

Skeletal animation via OpenMW NIF loader (covered by Gap 3 solution).
Facial animation via runtime texture swapping on the character mesh.
`rhubarb-lip-sync` (MIT licensed) for phoneme-to-mouth-shape mapping
from audio — matches the engine's three-shape approach (A, E, U
phoneme groups). Random animations and blinking as probabilistic Python
timers.

### Open questions requiring investigation

**OQ-8.1 — Phoneme-to-texture timing model**  
The engine drives `SpeakA`/`SpeakE`/`SpeakU` texture selection from
audio playback. The exact timing model — how many phoneme shapes are
evaluated per second, how transitions are smoothed — is engine-side.
Approach: instrument facial texture change events with audio timestamps
during a dialogue sequence to reverse-engineer the timing granularity.

**OQ-8.2 — Non-exclusive animation blending semantics**  
`SetNonExclusiveAnimation` implies animations can layer. The blending
weight and priority model between exclusive (full-body) and
non-exclusive (partial, e.g. head turn during idle) animations is
engine-side. Approach: instrument `TGAnimNode` state during sequences
where both animation types are active simultaneously.

**OQ-8.3 — MorphBody usage and scope**  
`CharacterClass.MorphBody()` suggests mesh morphing exists beyond
texture swapping. Used rarely in the visible source. Likely for species
variation or damage states. Approach: search all character creation
scripts for `MorphBody` calls, identify all usage contexts, assess
whether reimplementation is required for Phase 1.

**OQ-8.4 — Animation file catalogue completeness**  
Animations are loaded by string path from `data/animations/*.nif`. The
full catalogue of animation files is not in the SDK — only the scripts
referencing them. A complete reimplementation needs all animation assets.
Approach: extract all `LoadAnimation` call paths from the Python source
statically to build a complete asset manifest before attempting
reimplementation.

## Additional Discoveries

These patterns were not captured in the initial gap analysis but were found
during full SDK source review. None require a new gap — they refine the
implementation approach for existing gaps.

**ProcessWrapper pattern (affects Gap 4 and Gap 7)**  
`PythonMethodProcess` is a C++ object and cannot be pickled. Any mission
object that needs to survive save/load and owns a polling process must wrap
it in a plain Python class that stores the delay, priority, and function
name separately, then recreates the process on `__setstate__`. This pattern
appears in `Bridge/HelmMenuHandlers.py` and `Bridge/PowerDisplay.py` and
must be followed in any Phase 1 reimplementation of polling processes.

**Ship construction sequence (affects Gap 1 and Gap 2)**  
The full construction sequence is visible in `loadspacehelper.py:54–135`:
physics properties are set first (`SetMass`, `SetRotationalInertia` from
`GlobalPropertyTemplates.py`), then hardpoints are loaded via dynamic import
of `ships.Hardpoints.<shipname>`, then `SetupProperties()` is called to
finalize. This is the integration point between Gap 1 (Appc interface) and
Gap 2 (physics parameters) — the load helper is the seam.

**Weapon timing constants (affects Gap 2)**  
Weapon cooldown granularity is 0.05s minimum; typical recharge periods are
0.2–0.33s. These are hardcoded in the hardpoints files and are directly
usable in the Phase 1 implementation without instrumentation.

**Timer lifecycle management (affects Gap 7)**  
`MissionLib.py:141–162` shows that mission timers and episode timers are
tracked in separate lists and cleaned up on mission end via
`DeleteAllMissionTimers()` and `DeleteAllEpisodeTimers()`. The reimplemented
timer manager must support this two-tier lifecycle to avoid timer leaks
across mission boundaries.
